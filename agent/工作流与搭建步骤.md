# 知心陪练 · 扣子工作流设计与搭建步骤

> 双智能体在扣子上的落地方式 + 操作步骤。每步截图,直接用作过程性证明材料。

## 一、总体结构

```
用户 ──► 训练主控(智能体,多 Agent 模式)
          ├─ 案例智能体(扮演来访同学,读 case_script,维护信任值)
          ├─ 督导智能体(逐句评分 JSON + 会谈总结)
          └─ 工作流 × 3 + 数据库 × 2 + 知识库 × 1
```

扣子的「多 Agents 模式」允许一个应用内编排多个角色节点;若使用单智能体方案,也可以用工作流串联两次大模型节点(来访者节点 → 督导节点)实现同等效果——答辩时两种实现都讲得清。

## 二、数据库表

### session_log(会谈记录)
| 字段 | 类型 | 说明 |
|---|---|---|
| date | String | 2026-07-20 |
| case_id | String | c1~c4 |
| score | Integer | 百分制总分 |
| pass | Boolean | 是否通关 |
| refer_ok | Boolean | 危机案例转介是否正确(非危机案例为空) |
| dims | String | 五维得分 JSON |

### turn_log(逐句评分)
| 字段 | 类型 | 说明 |
|---|---|---|
| session_id | String | 关联会谈 |
| user_text | String | 用户原话 |
| dim | String | listen/empathy/question/boundary/refer |
| score | Integer | 0/1/2 |
| comment | String | 督导点评 |

## 三、工作流

### 工作流 1:start_case(开始会谈)
1. 用户选择案例(c1~c4,按 session_log 判断解锁进度)
2. 从知识库/变量读取对应 case_script,注入案例智能体
3. 案例智能体输出开场白,写入 session 上下文

### 工作流 2:turn_eval(每轮评分)
1. 用户发言后并行两路:
   - 案例智能体生成来访者回应(含〔信任:n〕标记,代码节点剥离标记后展示)
   - 督导智能体输出评分 JSON → 代码节点校验解析 → 写入 turn_log
2. 督导点评以"悄悄话"形式附在回应下方(可设开关:实时显示 / 仅结束后显示)

### 工作流 3:end_report(结束复盘)
1. 汇总 turn_log 五维得分,代码节点计算总分与通关判定(危机案例 refer_ok=false 一票否决)
2. 督导智能体生成总结报告(三句话复盘)
3. 写入 session_log,输出报告 + 下一关解锁提示

## 四、安全设置

- 案例智能体与督导智能体的 prompt 均含危机红线(见各自文件)
- 主控增加全局兜底:用户以本人身份表达真实自伤念头 → 立即暂停训练,输出 12355/12356 与求助引导(此逻辑同样写入人设最高优先级)
- 不开启联网搜索;知识库仅挂载自有文档
